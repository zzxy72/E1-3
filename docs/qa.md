# Q&A

## 1. 라벨 정규화(normalize_label)를 왜 별도 함수/모듈로 분리했는가?

라벨 정규화는 `expected`, 필터 키, 내부 판정 결과를 모두 같은 기준인 `Cross`, `X`로 맞추기 위해 분리했다.  
문자열 처리 로직을 JSON 로더, 판정 로직, 출력 로직에 각각 흩어 놓으면 `+`, `cross`, `x` 처리 방식이 서로 달라질 수 있다.  
별도 함수로 분리하면 라벨 해석 규칙이 한 곳에 모이기 때문에 코드 중복을 줄일 수 있다.  
요구사항 변경으로 새로운 표현이 추가되더라도 `normalize_label()`만 수정하면 전체 동작에 반영된다.  
또한 테스트도 쉬워진다. 예를 들어 `+ -> Cross`, `x -> X`, 잘못된 입력의 예외 발생 여부를 이 함수만 따로 검증할 수 있다.  
결과적으로 유지보수성, 일관성, 재사용성이 좋아지고, 문자열 불일치 때문에 PASS가 FAIL로 바뀌는 실수를 줄일 수 있다.

## 2. 패턴 크기가 N일 때 연산 횟수가 $N^2$으로 증가하며, 측정 결과가 $O(N^2)$ 경향을 보이는 이유는 무엇인가?

*(1) 연산 횟수가 $N^2$으로 증가하는 이유*  
MAC (Multiply-Accumulate) 연산을 수행하려면, 입력된 패턴과 기준 필터 배열의 **모든 위치(요소)를 빠짐없이 한 번씩 방문**하여 서로 곱하고 그 결과를 더해 나가야 합니다.  
가로 $\times$ 세로 크기가 $N$인 패턴은 기본적으로 요소가 $N \times N$, 즉 $N^2$개로 이루어져 있습니다. 예를 들어 3x3 크기면 9개, 13x13 크기면 169개, 25x25 크기면 625개의 요소가 존재합니다.  
따라서 점수를 도출해내기 위해서는 데이터 공간의 넓이에 해당하는 요소의 총개수와 동일하게 정확히 $N^2$번 곱셈 덧셈을 수행해야 하므로, 크기 $N$이 커질수록 연산 횟수는 $N^2$만큼 증가하게 됩니다.

*(2) 측정 속도가 $O(N^2)$ 경향을 보이는 이유 (시간 복잡도)*  
Big-O 표기법에서 $O(N^2)$은 "데이터 크기 N이 증가할 때 연산 소요 시간이나 자원 사용량이 N의 제곱에 비례하여 증가하는 경향"을 명확히 뜻합니다.
앞선 성능 분석표 결과를 보면:
- **3x3 ($N^2=9$)** 의 소요 시간: 약 0.005 ms
- **25x25 ($N^2=625$)** 의 소요 시간: 약 0.222 ms 
처럼 데이터 크기가 극단적으로 커질수록 걸리는 속도가 함께 폭발적으로 기하급수적 우상향 곡선을 그립니다.  
프로그램 코드적으로 살펴보더라도, $N^2$ 크기를 갖춘 1차원 데이터끼리 1:1로 매칭시키기 위한 순차적 단일 `for 루프(flat list 기준)`가 입력된 배열의 총 길이($N \times N$)만큼 정확하게 반복 수행됩니다. 
I/O 구간을 제외하고 순수하게 데이터 탐색 횟수에 의해서만 실행 시간이 좌우되기 때문에 이론(연산 횟수 $N^2$)과 실제 측정 시간 현상이 완벽하게 일치하는 $O(N^2)$ 경향성을 보이게 되는 것입니다.
