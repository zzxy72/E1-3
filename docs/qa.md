# Q&A

## 1. 라벨 정규화(normalize_label)를 왜 별도 함수/모듈로 분리했는가?

라벨 정규화는 `expected`, 필터 키, 내부 판정 결과를 모두 같은 기준인 `Cross`, `X`로 맞추기 위해 분리했다.  
문자열 처리 로직을 JSON 로더, 판정 로직, 출력 로직에 각각 흩어 놓으면 `+`, `cross`, `x` 처리 방식이 서로 달라질 수 있다.  
별도 함수로 분리하면 라벨 해석 규칙이 한 곳에 모이기 때문에 코드 중복을 줄일 수 있다.  
요구사항 변경으로 새로운 표현이 추가되더라도 `normalize_label()`만 수정하면 전체 동작에 반영된다.  
또한 테스트도 쉬워진다. 예를 들어 `+ -> Cross`, `x -> X`, 잘못된 입력의 예외 발생 여부를 이 함수만 따로 검증할 수 있다.  
결과적으로 유지보수성, 일관성, 재사용성이 좋아지고, 문자열 불일치 때문에 PASS가 FAIL로 바뀌는 실수를 줄일 수 있다.

## 2. MAC 점수가 높을수록 "더 유사하다"고 판단할 수 있는 이유를 설명할 수 있는가?

MAC 점수는 입력 패턴과 필터를 같은 위치끼리 곱해서 모두 더한 값이다.  
쉽게 말하면, 두 그림을 겹쳐 봤을 때 "같은 자리에 같은 모양이 얼마나 많이 있느냐"를 숫자로 세는 것이다.

예를 들어 십자가 모양 입력이 십자가 필터와 잘 맞으면, 1인 자리끼리 많이 겹치기 때문에 더 큰 점수가 나온다.  
반대로 X 필터와는 많이 어긋나므로 곱한 결과가 0이 많이 나오고, 점수가 낮아진다.

즉, MAC 점수가 높다는 것은 두 데이터가 더 많이 맞아떨어진다는 뜻이고, 그래서 "더 유사하다"고 판단할 수 있다.  
초보자 관점에서 보면, 이것은 정답 그림과 입력 그림을 비교할 때 "겹치는 부분이 많을수록 더 비슷하다"고 보는 것과 같다.

간단한 예시로 보면 다음과 같다.

* 완전히 같은 모양: 같은 위치에서 1이 많이 겹쳐서 점수가 큼
* 조금 다른 모양: 일부만 겹쳐서 점수가 중간 정도
* 전혀 다른 모양: 겹치는 부분이 적어서 점수가 작음

그래서 우리 프로그램은 두 필터의 점수를 비교해서 더 큰 쪽을 더 유사한 것으로 판정한다.

## 3. 패턴 크기가 N일 때 연산 횟수가 $N^2$으로 증가하며, 측정 결과가 $O(N^2)$ 경향을 보이는 이유는 무엇인가?

*(1) 연산 횟수가 $N^2$으로 증가하는 이유*  
MAC (Multiply-Accumulate) 연산을 수행하려면, 입력된 패턴과 기준 필터 배열의 **모든 위치(요소)를 빠짐없이 한 번씩 방문**하여 서로 곱하고 그 결과를 더해 나가야 합니다.  
가로 $\times$ 세로 크기가 $N$인 패턴은 기본적으로 요소가 $N \times N$, 즉 $N^2$개로 이루어져 있습니다. 예를 들어 3x3 크기면 9개, 13x13 크기면 169개, 25x25 크기면 625개의 요소가 존재합니다.  
따라서 점수를 도출해내기 위해서는 데이터 공간의 넓이에 해당하는 요소의 총개수와 동일하게 정확히 $N^2$번 곱셈 덧셈을 수행해야 하므로, 크기 $N$이 커질수록 연산 횟수는 $N^2$만큼 증가하게 됩니다.

*(2) 측정 속도가 $O(N^2)$ 경향을 보이는 이유 (시간 복잡도)*  
Big-O 표기법에서 $O(N^2)$은 "데이터 크기 N이 증가할 때 연산 소요 시간이나 자원 사용량이 N의 제곱에 비례하여 증가하는 경향"을 명확히 뜻합니다.
앞선 성능 분석표 결과를 보면:
- **3x3 ($N^2=9$)** 의 소요 시간: 약 0.005 ms
- **25x25 ($N^2=625$)** 의 소요 시간: 약 0.222 ms 
처럼 데이터 크기가 극단적으로 커질수록 걸리는 속도가 함께 폭발적으로 기하급수적 우상향 곡선을 그립니다.  
프로그램 코드적으로 살펴보더라도, $N^2$ 크기를 갖춘 1차원 데이터끼리 1:1로 매칭시키기 위한 순차적 단일 `for 루프(flat list 기준)`가 입력된 배열의 총 길이($N \times N$)만큼 정확하게 반복 수행됩니다. 
I/O 구간을 제외하고 순수하게 데이터 탐색 횟수에 의해서만 실행 시간이 좌우되기 때문에 이론(연산 횟수 $N^2$)과 실제 측정 시간 현상이 완벽하게 일치하는 $O(N^2)$ 경향성을 보이게 되는 것입니다.

## 4. 부동소수점 오차가 왜 발생하며, 비교 정책(epsilon)이 왜 필요한지 설명할 수 있는가?

부동소수점 오차는 실수 값을 이진수로 표현할 때 생긴다. `0.1`, `0.2`처럼 우리가 흔히 쓰는 소수는 2진법에서 정확히 표현되지 않는 경우가 많아서, 저장과 연산 과정에서 아주 작은 반올림 오차가 누적된다.  
MAC 연산은 이런 값을 여러 번 더하는 구조이므로, 같은 수식을 계산해도 연산 순서나 누적 방식에 따라 최종 점수가 미세하게 달라질 수 있다. 그래서 `score_a == score_b` 같은 정확 비교는 실무에서 안정적이지 않다.

이 문제를 막기 위해 비교 정책으로 `epsilon`을 사용한다. 즉, 두 점수의 차이가 충분히 작으면 같은 값으로 간주하는 방식이다.  
예를 들어 `abs(score_a - score_b) < 1e-9`이면 동점으로 처리하면, 계산 환경이나 반올림 차이 때문에 판정이 흔들리는 상황을 줄일 수 있다.

실무에서는 epsilon을 단순한 임시값이 아니라 운영 규칙으로 관리하는 것이 중요하다. 권장되는 정책은 다음과 같다.

* 비교 기준은 한 곳에서만 관리한다. 예: `COMPARE_EPSILON = 1e-9`
* 점수 출력값은 원값을 유지하되, 판정은 epsilon 기준으로만 수행한다.
* `|a-b| <= epsilon`이면 동점으로 처리하고, 그 외에는 큰 점수를 승자로 본다.
* epsilon 변경이 필요하면 코드, QA 문서, 테스트 케이스를 함께 갱신한다.
* 경계값 테스트를 별도로 둔다. 예: 차이가 `1e-10`, `1e-9`, `1e-8`인 경우를 모두 확인한다.

이 프로젝트처럼 입력 크기와 점수 범위가 제한된 환경에서는, 복잡한 상대오차보다 단순한 절대오차 정책이 더 적합하다.  
즉, "러프하게 같다고 볼 수 있는 범위"를 미리 정해 두고, 그 범위를 넘는 경우에만 다른 값으로 판정하는 방식이 운영상 가장 안정적이다. 이 정책은 모드 1의 동점 판정과 모드 2의 PASS/FAIL 판단을 일관되게 유지하는 데도 도움이 된다.

## 5. 동점 처리 규칙을 "mode1 vs mode2"로 다르게 둔 이유를 요구사항 관점에서 설명할 수 있는가?

모드 1과 모드 2는 "무엇을 보여줘야 하는지"가 다르기 때문에 동점 처리도 다르게 정했다.  
쉽게 말하면, 모드 1은 사람이 결과를 확인하는 화면이고, 모드 2는 정답과 비교해서 자동 채점하는 화면이다.

모드 1에서는 사용자가 직접 필터 A와 B를 보고 "어느 쪽이 더 비슷한지" 확인한다.  
이때 두 점수가 거의 같으면 억지로 A나 B를 고르는 것보다 `판정 불가`로 보여주는 편이 더 친절하다.  
왜냐하면 둘 중 하나를 무리하게 선택하면, 실제로는 차이가 거의 없는데도 틀린 것처럼 보일 수 있기 때문이다.

반면 모드 2는 `expected` 값이 있는 자동 검증 모드다.  
여기서는 프로그램이 낸 판정이 정답과 맞는지 PASS/FAIL로 판단해야 하므로, 동점이 나오면 `UNDECIDED`로 두고 실패로 집계한다.  
즉, "둘 다 비슷해서 못 고르겠다"는 결과는 자동 채점 기준에서는 정답 판정이 아니기 때문이다.

정리하면 다음과 같다.

* 모드 1: 사람이 보는 실습 화면이므로 동점이면 `판정 불가`
* 모드 2: 정답 비교용 자동 채점이므로 동점이면 `UNDECIDED` 후 FAIL

이렇게 나눈 이유는 요구사항의 목적이 서로 다르기 때문이다.  
모드 1은 학습과 확인이 목적이고, 모드 2는 데이터셋 검증과 테스트가 목적이다.  
따라서 같은 "동점"이라도, 사용자에게 보여주는 의미와 채점 기준을 구분해서 처리해야 요구사항에 맞고 결과도 혼동되지 않는다.
