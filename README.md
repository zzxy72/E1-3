# E1-3 Mini NPU Simulator

Mini NPU Simulator는 MAC(Multiply-Accumulate) 연산을 직접 구현해 Cross(십자가)와 X 패턴을 판별하는 Python 콘솔 애플리케이션이다.  
요구사항의 실행 모드와 성능 분석을 모두 포함하고, 보너스 과제인 패턴 생성기와 1차원 배열 최적화도 반영했다.

시스템 구조와 설계 의도는 [docs/system-design.md](docs/system-design.md)에 별도로 정리해 두었고, 이 README는 실행과 결과 확인 중심 문서로 유지한다.  
문서화된 QA 설명은 [docs/qa.md](docs/qa.md), 요구사항 원문과 체크리스트는 [요구사항/내용.md](요구사항/내용.md)와 [요구사항/체크.md](요구사항/체크.md)에서 확인할 수 있다.

## 실행 방법

### 요구 환경

- Python 3.8 이상
- 외부 라이브러리 사용 없음
- 표준 라이브러리만 사용

### 실행 명령

```bash
python main.py
```

### 모드 선택

프로그램 실행 후 아래 세 모드 중 하나를 선택한다.

1. 사용자 입력 `(3x3)`
2. `data.json` 분석
3. 패턴 자동 생성

### 클래스 중심 구조

- `Pattern`
  - 입력 패턴 데이터를 표현한다.
  - 크기, expected, case_id 같은 패턴 정보를 함께 가진다.

- `Filter`
  - Cross 또는 X 기준 필터를 표현한다.
  - 자신의 필터 값으로 패턴 점수를 계산하는 책임을 가진다.

- `MiniNPUSimulator`
  - 사용자 입력 모드, JSON 분석 모드, 자동 생성 모드의 전체 실행 흐름을 관리한다.
  - `Pattern`과 `Filter`를 조합해 판정과 성능 분석을 수행한다.

## 구현 요약

### 1. 라벨 정규화

프로그램 내부 표준 라벨은 `Cross`, `X` 두 개만 사용한다.

- `+` -> `Cross`
- `cross` -> `Cross`
- `x` -> `X`

이 정책은 JSON의 `expected` 값과 필터 이름을 같은 기준으로 비교하기 위해 필요하다.

### 2. MAC 연산

외부 라이브러리 없이 반복문으로만 구현했다.

```python
score = 0.0
for r in range(n):
    for c in range(n):
        score += pattern[r][c] * filt[r][c]
```

### 3. 동점 처리 정책

부동소수점 오차를 고려해 아래 기준을 적용했다.

```python
abs(score_cross - score_x) < 1e-9
```

위 조건이면 `UNDECIDED`로 판정한다.

### 4. 예외 처리

- 사용자 입력 모드:
  - 숫자 파싱 실패
  - 행/열 개수 불일치
  - 오류 메시지 출력 후 재입력 유도

- `data.json` 분석 모드:
  - 케이스별 스키마 오류
  - 크기 불일치
  - 필터 누락
  - 전체 프로그램 종료 없이 해당 케이스만 `FAIL`

## 재현성 확인

### 1. 사용자 입력 예시

아래 입력으로 확인했다.

- 필터 A

```text
0 1 0
1 1 1
0 1 0
```

- 필터 B

```text
1 0 1
0 1 0
1 0 1
```

- 패턴

```text
1 0 1
0 1 0
1 0 1
```

실행 결과:

- A 점수: `1.000000`
- B 점수: `5.000000`
- 연산 시간(평균/10회): 환경에 따라 달라짐
- 판정: `B`

### 2. 입력 오류 검증 예시

`3x3` 입력에서 `1 2`처럼 숫자 2개만 넣으면 아래 메시지가 출력된다.

```text
입력 형식 오류: 각 줄에 3개의 숫자를 공백으로 구분해 입력하세요.
```

### 3. JSON 오류 케이스 검증 예시

임시로 `size_5_99` 케이스에 `2x2` 입력을 넣어 실행했을 때 프로그램은 중단되지 않았고, 아래처럼 케이스 단위로 실패를 기록했다.

```text
size_5_99: FAIL (행 수 오류: 5개의 행이 필요합니다.)
```

## 결과 리포트

### PASS/FAIL 요약

기본 `data.json` 기준 실행 결과는 아래와 같다.

| 항목 | 값 |
| --- | ---: |
| 총 테스트 수 | 8 |
| 통과 수 | 6 |
| 실패 수 | 2 |

실패 케이스:

- `size_5_3`: 중심값만 1인 패턴으로 Cross/X 점수가 같아 `UNDECIDED`가 발생한 케이스
- `size_5_4`: Cross 패턴인데 expected를 `X`로 둔 일반 오답 케이스

### 성능 분석 결과

아래 값은 2026-04-16 로컬 실행에서 `10회 평균`으로 측정한 예시다.  
실행 환경에 따라 절대 시간은 달라질 수 있지만, 크기가 커질수록 증가하는 경향과 1D(flat) 방식의 비교 용도는 유지된다.

| 크기(NxN) | 방식 | 평균 시간(ms) | 연산 횟수(N^2) | 반복 횟수 |
| --- | --- | ---: | ---: | ---: |
| 3x3 | 2D | 0.005330 | 9 | 10 |
| 3x3 | 1D(flat) | 0.001744 | 9 | 10 |
| 5x5 | 2D | 0.005915 | 25 | 10 |
| 5x5 | 1D(flat) | 0.002470 | 25 | 10 |
| 13x13 | 2D | 0.022403 | 169 | 10 |
| 13x13 | 1D(flat) | 0.013433 | 169 | 10 |
| 25x25 | 2D | 0.098161 | 625 | 10 |
| 25x25 | 1D(flat) | 0.064751 | 625 | 10 |

### 실패 원인 분석

이번 기본 데이터셋에서는 총 8건 중 2건이 FAIL로 기록되었다.  
첫 번째 실패 케이스 `size_5_3`은 의도적으로 중심값만 1인 패턴을 넣어 Cross 점수와 X 점수가 모두 1이 되도록 만든 사례다.  
이 케이스는 단순 비교라면 한쪽으로 잘못 판정될 수 있지만, 현재 구현은 `abs(score_a - score_b) < 1e-9` 허용오차 규칙을 적용해 `UNDECIDED`로 처리한다.  
그리고 expected를 `X`로 두었기 때문에 최종 결과는 FAIL이 된다.  
이 실패는 로직 오류가 아니라 동점 처리 정책이 정상 작동했음을 보여주는 검증용 실패다.  
두 번째 실패 케이스 `size_5_4`는 실제 입력 패턴은 Cross인데 expected를 `X`로 둔 일반 오답 케이스다.  
이 경우 Cross 점수가 더 높기 때문에 프로그램은 `Cross`로 판정하고, expected와 다르므로 FAIL을 출력한다.  
이 실패는 PASS/FAIL 비교 로직이 정상적으로 작동하는지 보여 주는 검증용 데이터다.  
한편 나머지 PASS 케이스들은 라벨 정규화 정책 덕분에 `+`, `cross`, `x` 같은 서로 다른 표현을 내부 표준 라벨 `Cross`, `X`로 안전하게 통일한 결과다.  
또한 JSON 키에서 크기를 읽어 올바른 필터를 선택하고, 크기 검증과 케이스 단위 실패 처리를 적용했기 때문에 일부 데이터가 잘못돼도 전체 실행은 유지된다.  
즉 이번 데이터셋은 PASS만 모아 둔 구성이 아니라, 정상 분류, 동점 기반 FAIL, 일반 오답 FAIL을 모두 포함해 요구사항 검증에 더 적합한 형태로 구현함.

### 시간 복잡도 분석

MAC 연산은 패턴과 필터의 같은 위치 원소를 한 번씩 곱하고 누적합을 계산한다.  
즉 `N x N` 입력에 대해 각 원소를 정확히 한 번씩 방문하므로 기본 연산 수는 `N^2`에 비례한다.  
실제 표에서도 3x3은 9회, 5x5는 25회, 13x13은 169회, 25x25는 625회의 위치 비교가 필요하다.  
측정 시간 역시 크기가 커질수록 전반적으로 증가했다.  
3x3에서는 2D 방식이 `0.003575 ms`, 25x25에서는 `0.073775 ms`로 커졌고, 1D(flat)도 같은 증가 추세를 보였다.  
절대 시간은 운영체제 스케줄링, 인터프리터 상태, CPU 부하에 따라 흔들릴 수 있다.  
그래서 한 번만 재지 않고 10회 평균을 사용해 비교 기준을 맞췄다.  
1D(flat) 최적화는 메모리 접근을 단순화해 일부 구간에서 더 낮은 측정값을 보여 주었다.  
다만 이 최적화가 시간 복잡도 자체를 바꾸는 것은 아니고, 여전히 모든 원소를 한 번씩 순회하므로 빅오는 O(N^2)다.  
따라서 이 프로젝트의 핵심 MAC 알고리즘은 요구사항대로 O(N^2)이며, 성능 표는 그 근거를 실제 실행 값으로 보강하는 역할을 한다.

## 보너스 과제 요약

- 패턴 생성기: 크기 `N`의 Cross/X 패턴을 자동 생성해 모드 1과 성능 분석에 재사용했다.
- 1차원 배열 최적화: 2차원 데이터를 `flat` 형태로 바꿔 MAC 경로를 단순화하고, 실제 측정으로 효과를 확인했다.
